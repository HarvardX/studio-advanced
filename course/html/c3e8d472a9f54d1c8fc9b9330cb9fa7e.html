Limitations of the edX Mobile App

EdX provides a good set of guidelines for designing for mobile users. However, we'd also like to add a few of our own, and highlight some pain points for courses with a large number of mobile users.

Supported Components

EdX's mobile app delivers video and HTML acceptably well. However, is very limited in its use of problems and tools. You can see the full up-to-date list at the official edX documentation, but as of 2019 the list of supported tools and activities is as follows:

  • Checkbox Problem
  • Drag and Drop Problem
  • Dropdown Problem
  • Math Expression Input Problems
  • Multiple Choice Problem
  • Numerical Input Problem
  • Text Input Problem

For items not on this list, the app will link the user to a web view of the course in a mobile browser. It works, but it's definitely not an ideal experience.

The mobile app also doesn't support certain forum features, like blackout dates.

Supported Levels of Hierarchy

EdX normally has four levels of hierarchy within a course: sections (the top level) contain subsections, which contain units, which contain components (the bottom level).

However, that's not always the case. When you use A/B testing, conditionals, and other "container" blocks, edX gains a fifth level of hierarchy. The mobile app does not support this fifth level well, and learners will see nonsensical titles for their components, with a "Group ID" label.

"Boxed-in" Components

One of the biggest limitations of edX's mobile app, as compared with a browser experience, is that only one component is displayed at a time. This can split introductions away from videos, and problem details away from the problems themselves. Interconnected problem sets don't all show up at once; learners see just a bit at a time.

This makes things especially tricky when it comes to Javascript and CSS. They appear only on the component where they are included, and do not affect other components in that unit (as they do in a browser). Many of our Javascript approaches are designed to "fail gracefully" when the script is boxed into a single component. When you build your own JS, you should consider what the student experience will be in these situations.

Recommendations

Honestly, even given all these issues with the mobile app, we still do not suggest building your course to fit completely within its limitations. Instead, we recommend building the course with all the features you'd like to have, and planning for a browser-based experience. There are three reasons for this:

  1. All mobile users have a browser available in addition to the app.
  2. Most mobile browsers have excellent Javascript support, and will handle the edX site and special tools quite well.
  3. EdX has not put substantial support toward the mobile app, and many of their actions (like moving to reactive layouts) imply that they may shift toward a browser-based approach to mobile users rather than an app-based approach.

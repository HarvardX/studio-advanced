The ORA Toolbox

Hopefully that was a helpful exercise for you. There are a few other fun toys that we wanted to point out - but some of them come with caveats.

It's not Peer Instruction

That's a different thing - see the UBC Peer Instruction tool docs instead.

What to Expect from your Learners

Research on peer assessments has consistently indicated that:

  1. Learners do take them seriously, and most will try to grade each other appropriately.
  2. People learn as much from performing their own self-assessment as they do from seeing their peers' assessment of them, perhaps more.

What we're trying to say here is that ORA is just as valid an assessment tool as any other! However, there are few things to watch out for:

  • Don't Rush the Rubric. The quality and specificity of the rubric is far more important than any supposed skill level of the learners. A good rubric asks as little from the learners as possible, and gives them ways to be certain that they are grading the other person correctly. Take the extra time to write it carefully and test test test before you release it into the wild.
  • Don't Rely on Open Feedback. The amount of free-response feedback is typically less than one tweet's worth. Once again, count on the rubric, not on human expertise.

The Date Trick

Do you have a self-paced course and want to use ORA, but don't want it to show a date? Set your due dates to 9999-01-01, and the system will stop saying things like "due in 980 years".

Uploading Files

You can enable file uploads for Open Response Assessments. Perhaps you've got a course where you're teaching a specific software tool and you want people to be able to see each others' files. Maybe it's a drawing course, and they're sharing photos of their work.

Watch out for:

  • Accessibility issues can crop up easily with certain file types. If you're teaching a drawing course then visual media is obviously a must, but do try to take into account the needs of all your learners.
  • Not everyone has Word. Strange but true - there are only a few really "universal" file types out there. Remember also that more people can read certain files than write them, so if you require submissions in PDF format you might end up excluding certain people. We recommend suggesting that your students upload .txt or .rtf files for text, and .png or .jpg for images.
  • Code? If you let your learners upload code in a computer science course, someone else might run that code, even if you tell them to evaluate it "by eye". That's a serious security risk that you're putting on your students. We recommend against using ORA this way.

What's the purpose of cohorts?

Cohorts first came about through a desire to run content simultaneously for the masses and for credit-earning students on campus, with the latter group having access to specific additional content. As the platform developed, new use cases also developed in parallel, such as running smaller classes (on edge.edX) with variable pathways, conditional release and study groups with group projects. Beyond this, cohorts are most often used within A-B testing for research or as a method to foster greater community within discussion boards.

Automatic vs. Manual

There are two cohort assignment methods: automatic or manual.

  • Automatic: you decide the number of cohort groups and users are randomly assigned for you by the platform
  • Manual: you decide the number and allocate users directly yourself via individual or batch upload (more common on small, private courses)

In most cases, the default will be to use ‘automatic’ user allocation. This is by far the most straightforward method and ensures no user is ‘accidentally missed’, as every user is assigned to a cohort group upon their first access to the course site, once the course is live.

Manual cohorts can be a little more fiddly to set up, as learners must be enrolled on the course before they can be assigned to a group. What this means in practice: waiting for everyone to sign up, hassling people a lot to sign up or enrolling them onto edX and/or the course before. Speculative pre-loading of user information is not possible.

It is really important to decide in advance which allocation method you are going to use as the process is a little more time intensive for manual methods. As a result, before enabling cohorts on your course, it is highly recommended to develop a ‘Cohort Strategy’. 

A Cohort Strategy

Basically what it says on the tin: a plan for using cohorts on the course

If you go in and start making cohorts without a clear strategy, it can get a bit messy to rectify. The cohort setup process is very logical but needs to be tackled in a sequential order.

The questions below might provide a helpful guide:

Developing a strategy

  1. What is the purpose of cohorts for your course?
    • Research, e.g. comparison between variables
    • Community, e.g. study groups
    • Restricted access, e.g. student access to additional materials on an open course
  2. Is there a default group?
  3. How many cohorts do you need?
    • What number of unique groups are required?
    • Is there an easily identifiable reference to use for each group?
  4. How do you want users allocated?
    • Random allocation (automatic allocation) 
    • Via specific characteristics (manual allocation)
  5. What is the difference between each group? 
    • Do they need access to different materials?
  6. Do any groups need to be able to interact in specific places, e.g. combined discussions?

Once you have a clear idea of what you want, you’ll need to consider how this translates into platform setup.

Platform considerations

There are only two platform features that can be manipulated via cohorts: content (units and components) and discussion boards.

Specific content, such as units (full pages) or components within a unit can be assigned ‘restrictive view’. Sections and subsections can not be restricted. This means your course architecture will remain the same for everyone. However, there is nothing stopping you from making multiple copies of a unit, manipulating particular variables in that unit, then associating each unit copy with a different cohort to provide a standardised front end (if important).

Content view differences are changed through ‘content groups’. After creating a content group, you can assign a cohort (or multiple) to view it.

Each cohort gets its own discussion board area allocated by default. However, any discussion boards featured in open units will be accessible to everyone. If you are planning covert A-B testing, open discussions may need consideration.

Learners can only be in one cohort - changing their cohort allocation will remove them from their first allocation, even if content or discussions have been engaged with. This is really important to remember as changing allocation is very risky if the course has already gone live. 

Manual Cohorting Tip

If using cohorts, consider a closed enrollment period that ends prior to launch of the course, to ensure all learners can be accounted for and provide a small grace period for any last-minute allocation changes. This is particularly useful if you are running A-B testing with manual cohorts - final checks are always useful!

Default Cohorts

If no automated cohorts exist, the system will create a ‘default’ cohort and assign new learners to this group. To remove ‘default’ or reallocate to a different cohort, you would need to use the ‘manual’ allocation method. Check, check and check again that your automatic allocations are setup correctly before going live with a large open course.

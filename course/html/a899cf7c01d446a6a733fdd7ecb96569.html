How to spot rogue css changes

Not all changes made through CSS will be as dramatic as our examples. Some changes can also have unintended consequences on other parts of the page, in subtle, almost unnoticeable ways. However, there are some typical clues visible on platform to help you troubleshoot.

If you are adding new HTML to your course, you should always test it out on one page before rolling out across the course. The more complex the HTML, the more testing required. 

These tell tale signs are a good indication something needs to be toned back or removed:

CSS Checklist

In preview and live - remember to check both in separate tabs and toggle between to compare

  • Top site navigation bar changes - did the edX logo move? Is the happy little edX icon or your profile picture distorted?
  • Loading time - is the page having difficulty loading or not loading at all?
  • Unit navigation - are you able to press ‘next’ or select the next unit logo just above the main text body? 
  • Site look and feel - is the text suddenly bigger/smaller than normal? Is the displaying font different?
  • Did anything jump?

It’s common for changes to only be noticed once added to the platform. CSS viewers are great but give a clinical view of code - in a viewer, the code is not having to compete with all the other HTML built into the platform, so make sure you upload, check and update your code regularly during the development phase.

What to do if you spot rogue changes

When adding new CSS, check your page preview and live modes regularly to test implementation is as intended. If you notice something odd, go back to the last saved version of your CSS and focus on the most recent change. Do any elements need closing? What was the thing that changed on platform - does that provide a clue for your code?

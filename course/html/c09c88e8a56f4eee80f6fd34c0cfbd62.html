Why Make a New Problem Type?

Let's start this section with a warning:

Warning Workload Hazard

Only create a custom JS problem if you're certain that you can't do what you want with existing problem types. When you do build one, spend some extra time considering what small tweaks and options might make it more reusable for multiple situations. It's a lot of work, but no one wants to build a dozen new bespoke problems that will never be used again. You're not running a hipster assessment shop.

Now that we have that out of the way...

The great thing about JSInput problems is that you can do anything you want to code with them, within certain limitations (see sidebar). You decide what the learners do, and you decide how they're graded. That's a very wide-open field, so let's give a few examples:

Limitations of JSInput Problems

  1. No inter-student communication. Learners can only see their own work. You can't pass data from one learner to another.
  2. No inter-problem communication. Other assessments will not have access to a record of what a learner did here. You can't pass data from one assignment to another.
  • First, of course, there are the ones we've already built: matching exercises, ordering problems, numerical or time-based range sliders, text boxes that just log what the student types, and quick choose-your-own-adventure problems. All of these are at Stanford's JSInput Repository and free for you to use.
  • You can build a problem type that edX doesn't offer. Need a chemical bond checker? Write it! Want tabular input for data instead of a standard numerical problem? Go for it!
  • You can let students upload a simple file and parse it with Javascript, either analyzing it there in the browser or passing it to Python for grading. (Don't pass anything big. Be nice to your server.)
  • You could take an existing javascript-based interactive or game and wrap it in a JSInput problem so that you can grade what the learner does with it.
  • Similarly, you could create something that's less of a skill assessment and more of an interactive experience, which grades learners on participation rather than correctness.
  • You can use it to build familiarity with a tool. Is there an interactive that you use throughout your course? Wrap it in a JSInput problem and have your learners play with it, only giving full credit once they've tried out every single important feature.

Clearly there are a ton of options. Before you pick out exactly what you want to do, we recommend building a really simple problem. The experience you get with that will be essential when it comes to making the thing you really want to create.

Use separate lines

It might seem obvious, but just like writing a sentence, your code needs punctuation. It is a language, after all!

Breaking up content so it doesn't all form one massive block of text with code embedded will certainly help you find things and make improvements, above just looking nice! Think about your future self.

Remove duplicate references

Screenreaders (like VoiceOver, JAWS, and NVDA) read additional information, such as alt-text, aloud to the user. This might seem obvious, but how often have you taken the easy route on alt-text or references and just simply copied the suceeding text so there is something in the space? Hopefully the answer is 'never,' but no one is perfect. Getting into the habit of adding meaningful information for users in alt-text and title-text will not go unappreciated.

Reference link content, not urls

Another good habit to get into is to provide meaningful (but short) references for links. Rather than just referencing a link as 'click here' make reference to the information it will point to, e.g. 'Further Thermodynamics reading'.

If a url is referenced in a <a></a> a screenreader will pick up on this being a link and will say 'link' before the reference text. However, if you include the url as the reference, the screenreader will try and say this... often rather painfully letter-by-letter! Referencing that it is a link is also redundant, so you don't need to put that in the text.

Here's a before and after:

<a href="http://www.google.com" target="_blank">www.google.com</a>
<a href="http://www.google.com" target="_blank">Google search engine</a>

It might not seem like a lot but it makes a huge difference.

Figure containers for images

On the front view, image files displayed as <img> and <figure> can yield almost identical results but the code is very different. By default, edX will include an image file without any <figure> references, but if you want to add a caption, link to the image files or make consistency changes to all your images using CSS, a <figure> container for <img> files is highly recommended.

The best thing about using <figure> is the ability to connect these containers to your CSS. This allows you to cascade changes across all pages very easily, rather than having to manually change each image individually.

For example, a client decided half-way through build they wanted a grey border around all images and for images to appear 40% width when left or right aligned - this would have been a nightmare on a course with ~50 images in-line, but as each was configured as a <figure> we made the change to the css file directly and everything changed instantly. (Obviously, we could have said no... but whereâ€™s the fun in that!)

It is possible to change every <img> in your your course through CSS without using <figure> but it's pretty risky as EVERY <img> would be affected. Figure containers allow a little buffer and ease to add captions without the use of too many <div> or <br>.

Let's say you wanted all <img> to be width='80%' unless they have a caption in a <figure>, when they should display at width='100%'. If you changed your CSS to influence all <img> references, when it appears in a figure container you would see an 80% width rather than 100% as the page will render both 'required' widths in the image reference.

Finally, <figure> allows you to add good practice such as a link to the image file in a new tab and captions
